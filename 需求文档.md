**npu api alignment requirements v3**

**NPU PyTorch API 对齐看板**

需求与功能设计

v0.3 状态：草稿 2026-04-11

**1. 项目背景**

torch\_npu 作为 PyTorch 的 NPU 后端实现，需要将大量 PyTorch 原生 API 适配到昇腾 NPU 硬件上。当前各 API 与 CUDA 后端的对齐情况缺乏统一的可视化展示和管理，导致：

* 开发者难以快速查询某个 API 的支持状态和测试覆盖情况
* 测试团队缺少全局视角来评估覆盖率和质量
* 测试数据散落在各处，缺乏统一的数据源和可溯源性

本项目目标：建设一个 Web 平台，以 API 为核心维度，展示 NPU 后端与 CUDA 的对齐情况，覆盖功能、精度、内存、确定性四个测试维度。

**2. 用户与场景**

|  |  |
| --- | --- |
| **角色** | **核心场景** |
| 内部开发者 | 查询某个 API 的对齐状态，查看已测维度和具体用例结果，确认是否可以安全使用 |
| 测试工程师 | 了解哪些 API 需要补充测试，录入或更新测试结果，查看各维度覆盖率 |
| 项目管理者 | 查看全局对齐率、各模块进度、趋势变化 |
| 外部开发者 | 输入 GitHub repo 链接，查看该项目用到了哪些 API、对齐情况如何、torch\_npu 对该项目的支持程度 |

**3. 核心数据模型**

**3.1 测试维度定义**

每个 API 的对齐情况从四个维度衡量：

|  |  |
| --- | --- |
| **维度** | **含义** |
| 一致性 | 在以下四个维度上与 CUDA 保持一致 |
| 功能 | NPU 上的计算结果与 CUDA 一致，覆盖各 dtype、shape、边界条件。  **用例来源分为两类：**  • 社区用例：PyTorch 官方自带的测试用例，在 NPU 上复跑验证  • torch\_npu 用例：团队额外补充的测试用例，覆盖社区用例未覆盖的场景 |
| 精度 | 数值误差在可接受范围内，特别关注 fp16 / bf16。  **用例覆盖程度可区分多个层次（后续可细化）：**  • ATK 覆盖：通过 torch\_npu 的 ATK 工具自动生成的用例，覆盖最全面  • UT 覆盖：开发者手写的单元测试用例  • 其他覆盖方式待后续定义 |
| 内存 | 内存分配与释放模式与 CUDA 一致，无内存泄漏 |
| 确定性 | 相同输入多次运行结果一致，或与 CUDA 的非确定性行为一致 |

**3.2 对齐状态定义**

每个 API 的每个维度不一定能完全与 CUDA 一致，需要更细粒度的状态来描述对齐程度：

|  |  |  |
| --- | --- | --- |
| **状态** | **含义** | **示例** |
| 完全对齐 | 与 CUDA 行为完全一致 | 所有测试用例均通过，结果无差异 |
| 有差异（已评审） | 存在差异但经评审确认可接受，不会再修复 | 如：因硬件限制导致精度差异在 1e-3 内；内存用量比 GPU 高 50% 但确认为架构差异不再优化 |
| 有差异（待修复） | 存在差异，计划修复 | 如：某 dtype 下结果不一致，已提 issue 跟踪 |
| 不支持 | 当前未实现或确认不会支持 | 如：硬件不具备对应能力，明确不支持 |
| 未测试 | 尚未进行测试 | 无测试数据 |

**「有差异（已评审）」记录要求：**

* 必须附带评审结论说明（为什么接受这个差异）
* 必须记录评审人和评审时间
* 可选记录具体差异数据（如精度误差范围、内存比例等）

**3.3 API 对齐记录**

以「一个 API + 一个测试维度」为最小粒度：

|  |  |  |  |
| --- | --- | --- | --- |
| **字段** | **类型** | **示例** | **说明** |
| api\_name | string | Tensor.to | PyTorch API 完整名称 |
| api\_module | string | torch.Tensor | 所属模块，用于分类筛选 |
| api\_level | string | L0 | L0/L1/L2/待定级 |
| dimension | enum | 功能 | 功能 / 精度 / 内存 / 确定性 |
| status | enum | 完全对齐 | 见 3.2 对齐状态定义 |
| deviation\_note | string? | 见下方说明 | 差异说明（状态为「有差异」时必填） |
| reviewer | string? | list | 评审人（状态为「已评审」时必填） |
| reviewed\_at | datetime? | 2026-04-01 | 评审时间 |
| test\_cases | json[] | 见 3.4 | 关联的测试用例列表 |
| data\_source | enum | manual | manual / ci / import |
| source\_url | string? | CI job URL | 可溯源链接 |
| updated\_by | string | zhangsan | 最后更新人 |
| updated\_at | datetime | 2026-04-11 | 最后更新时间 |

**3.4 测试用例**

每条对齐记录可关联多个测试用例：

|  |  |  |
| --- | --- | --- |
| **字段** | **类型** | **说明** |
| case\_name | string | 用例名称，如 test\_to\_float16\_contiguous |
| case\_source | enum | 社区用例 / torch\_npu 用例（仅功能维度） |
| description | string? | 用例描述，说明测的是什么 |
| result | enum | pass / fail / error / skip |
| error\_message | string? | 失败时的错误信息 |
| code\_snippet | string? | 测试代码片段（可选） |
| run\_at | datetime | 用例执行时间 |

**4. 功能需求**

**4.1 API 搜索与详情页**

**优先级：P0** 用户能快速查询任意 API 的对齐状态和测试详情。

**搜索能力**

* 支持按 API 名称模糊搜索（输入 to 可匹配 Tensor.to、Tensor.to\_sparse 等）
* 支持按模块筛选（如 torch.Tensor、torch.nn.functional）
* 搜索结果列表显示 API 名称、模块、四个维度状态摘要

**API 详情页**

* API 基本信息：名称、所属模块、PyTorch 官方文档链接
* 四个维度的状态卡片（功能 / 精度 / 内存 / 确定性），每个卡片显示对齐状态和用例通过率
* 若状态为「有差异（已评审）」，卡片上突出显示差异说明、
* 评审人、评审时间
* 功能维度下区分显示社区用例和 torch\_npu 用例的通过情况
* 每个维度下可展开查看具体测试用例列表：用例名、结果、错误信息、代码片段
* 数据来源标识和可溯源链接

**示例场景**

开发者搜索 Tensor.to → 进入详情页 → 看到功能维度「完全对齐」，精度维度「有差异（已评审）」并显示“fp16 下 atol=1e-3 内，硬件限制不再优化”，内存维度「未测试」，确定性维度「完全对齐」。

**4.2 全局看板**

**优先级：P0** 提供全局视角的对齐状态总览。

**核心指标区**

* 总 API 数量、各状态（完全对齐 / 有差异 / 未测试）数量
* 四个维度分别的覆盖率

**API 列表**

* 表格形式，每行：API 名称、模块、四个维度状态图标、最后更新时间
* 支持按模块、状态、维度筛选
* 支持按名称、对齐率、更新时间排序
* 点击某行跳转到 API 详情页

**4.3 Diff 对比视图**

**优先级：P1** 支持不同时间点或不同版本之间的对齐状态对比。

* 选择两个时间点（或两个版本快照），以 diff 形式展示对齐状态变化
* 突出显示：新增对齐的 API、状态发生变化的 API、新增回归的 API
* 可用于版本发布前的对齐情况审查，或评估一段时间内的进展

**4.4 Repo 兼容性分析**

**优先级：P1** 输入 GitHub repo 链接，评估 torch\_npu 对该项目的支持程度。

**流程**

* 用户输入 GitHub repo URL
* 系统克隆并扫描仓库代码，提取其中使用的 PyTorch API 列表
* 与本平台对齐数据交叉匹配，生成兼容性报告

**兼容性报告内容**

* 该 repo 用到的 API 总数
* 各状态分布（完全对齐 / 有差异 / 不支持 / 未测试）
* 整体支持度评分（如百分比或等级）
* 风险 API 列表（不支持或有差异的 API 及其在代码中的使用位置）
* 支持导出报告为 PDF 或 JSON

**4.5 数据导入**

**优先级：P0** 支持初始数据的批量录入。

* 支持通过 CSV 或 JSON 文件批量导入对齐数据
* 导入时做格式校验（字段缺失、枚举值非法等）
* 导入完成后显示结果摘要：成功数 / 失败数 / 跳过数
* 支持重复导入（相同 API + 维度的记录覆盖更新）

**4.6 人工编辑后台**

**优先级：P1** 支持对单条记录的增删改查。

* 提供后台管理界面，支持新增、编辑、删除对齐记录
* 支持在某个维度下新增、编辑、删除测试用例
* 支持填写差异说明和评审信息
* 编辑时自动记录操作人和时间戳
* 操作日志可查询

**4.7 CI 上报接口**

**优先级：P1** 供 CI 系统自动上报测试结果。

* 提供 RESTful API，测试流水线跑完后自动调用
* 支持幂等操作（相同 API + 维度重复上报应覆盖而非重复创建）
* 上报数据自动标记 data\_source = ci，并要求附带 source\_url
* 接口鉴权机制待定（token / API key）

**4.8 数据导出**

**优先级：P1** 支持将当前视图数据导出。

* 看板或筛选结果可导出为 CSV / JSON
* 导出内容包含完整的对齐记录与用例信息

**4.9 趋势与统计**

**优先级：P2** 提供对齐进度的历史趋势视图。

* 对齐率随时间的变化趋势图
* 按模块的对齐情况分布图
* 最近 N 天新增或变更的记录流

**5. 数据流**

数据进入系统有三个通道，按自动化程度递进：

|  |  |  |  |
| --- | --- | --- | --- |
| **通道** | **触发方式** | **data\_source** | **适用场景** |
| 初始导入 | 上传 CSV/JSON 文件 | import | 一次性录入历史数据 |
| 人工维护 | 后台界面操作 | manual | 日常更新和修正 |
| CI 自动上报 | 流水线调用 API | ci | 自动化目标态 |

所有通道写入的数据均带有来源标识和时间戳，确保可溯源。CI 通道必须附带 source\_url。

**6. 优先级总览**

|  |  |  |
| --- | --- | --- |
| **级别** | **内容** | **目标** |
| **P0** | API 搜索详情页 + 全局看板 + 数据导入 | MVP：导入数据并查看对齐情况 |
| **P1** | 编辑后台 + CI 上报 + Diff 对比 + Repo 分析 + 导出 | 数据可维护，支持外部分析 |
| **P2** | 趋势图表 + 模块分布 + 变更流 | 数据洞察，辅助决策 |
